# 本地 Coding Agent 执行提示词（完整版本）

你是一名资深全栈工程师与文档自动化系统架构师。请基于当前提供的模板、脚本、示例 JSON 和说明文档，继续设计并实现一个“项目申报书自动生成系统”的下一阶段方案。你的目标不是只修补单个脚本，而是把现有原型整理成**可维护、可扩展、可验证**的本地工程。

## 一、项目目标

当前已经有一套原型能力：
- 使用 Word `.docx` 作为主模板
- 使用 `docxtpl + python-docx` 渲染内容
- 支持将“研究内容”整块作为 `subdoc` 插入主模板
- 已有基础版与进阶版 JSON 示例、渲染脚本、模板升级指南
- 已经考虑了：
  - 多个章节 subdoc
  - 多个附件
  - 可选章节开关
  - 表格、图片、图题、图例动态插入

请在此基础上，把系统设计成一个适合长期演进的本地项目。最终目标是：

1. 让用户基于不同申报书模板快速生成成品 Word 文档。
2. 支持复杂正文混排：多段文字、表格、图片、图题、图例、分页符。
3. 支持多个章节独立控制、多个附件统一生成。
4. 支持后续接入大模型，只要求大模型输出结构化 JSON，而不是直接输出 Word 富文本。
5. 支持本地验证与调试，便于人工检查生成结果。

## 三、你的总体任务

请围绕下面四个层面，继续做设计与实现。

### 1. 工程结构整理
将现有零散原型整理成一个清晰工程，例如：

- `templates/`：放主模板、示例模板
- `data/`：放示例 JSON、测试输入
- `scripts/`：放渲染脚本、命令行入口
- `docs/`：放说明、规范、升级指南
- `output/`：放生成结果
- `assets/`：放示例图片、附件素材
- `src/`：如果后续需要抽象为模块，可将渲染器拆入此目录

要求：
- 命名统一
- 目录职责清晰
- 便于后续增加模板和脚本

### 2. 渲染架构优化
将当前脚本优化成更清晰的架构。建议但不限于：

- `TemplateRenderer`
- `SectionBuilder`
- `SubdocBuilder`
- `AppendixBuilder`
- `BlockRenderer`
- `PayloadValidator`

要求：
- 结构清晰，函数职责单一
- 便于扩展新的 block 类型
- 出错时能给出明确报错信息

### 3. 数据结构规范化
完善 JSON schema / 数据约定。

至少要覆盖：
- 固定字段（项目名称、单位、负责人等）
- `sections`
- `appendices`
- `enabled`
- `order`
- `blocks`
- 各 block 类型字段要求

现有 block 类型包括：
- `heading`
- `paragraph`
- `bullet_list`
- `numbered_list`
- `table`
- `image`
- `page_break`

希望你进一步评估是否要支持：
- `quote`
- `note`
- `two_images_row`
- `appendix_table`
- `toc_placeholder`

### 4. 模板适配机制
设计一套**模板适配规则**，让不同单位、不同项目模板都能复用同一个引擎。

至少考虑：
- 占位符命名规范
- 样式命名规范
- 多章节 subdoc 的模板写法
- 附件总区块写法
- 条件显示章节的模板写法
- 模板样式缺失时的报错机制

## 四、你需要重点解决的问题

### 问题 1：复杂章节混排
像“研究内容”“研究基础”“实施方案”这类章节，里面可能同时出现：
- 二级标题
- 多段正文
- 项目符号列表
- 动态表格
- 图片
- 图题
- 图例
- 分页符

请继续沿用并完善“整章 subdoc”的思路，不要退回到在主模板中塞大量复杂 Jinja 标签。

### 问题 2：附件机制
附件可能有两种情况：
1. 每个附件有独立标题与独立内容
2. 多个附件自动汇总到“附件”章节中

请实现一种统一设计，使两种方式都能支持。

### 问题 3：可选章节开关
每个章节和附件都可能：
- 显示
- 隐藏
- 调整顺序

请设计一套清晰的 payload 结构和模板约定，让这些开关逻辑稳定可控。

### 问题 4：样式缺失问题
因为 `python-docx` 在应用不存在的样式时不会显式报错，所以请增加显式检查：
- 检查模板是否存在所需样式
- 如果缺失，应在生成前提示哪个样式不存在
- 不要让用户在生成后才发现 Word 格式失控

### 问题 5：本地调试体验
请设计一个本地调试流程，至少包括：
- 输入一个 JSON 示例
- 渲染出 docx
- 将结果输出到 `output/`
- 记录日志
- 给出最基本的异常信息

如果条件允许，可以再增加：
- 渲染结果快照对比
- 生成检查清单
- 缺失图片告警

## 六、技术要求

- 优先使用 Python
- 核心依赖可继续使用：
  - `docxtpl`
  - `python-docx`
- 代码要清晰、可维护、带必要注释
- 不要把所有逻辑堆在一个脚本里
- 不要让大模型直接参与 Word 格式控制
- 要把“样式控制”与“内容生成”分离

## 七、设计原则

请严格遵守以下原则：

1. **固定框架留在主模板，复杂正文整章做成 subdoc。**
2. **大模型只输出结构化 JSON，不输出 Word 富文本。**
3. **模板负责样式，程序负责内容插入。**
4. **能用统一样式解决的，不要用 run 级格式硬改。**
5. **复杂表头尽量固定在模板中，动态部分优先限制为表体。**
6. **图题、表题、图例都作为独立段落处理。**
7. **要优先考虑后续扩展多个模板，而不是只适配当前示例。**
